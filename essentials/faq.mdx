## Frequently Asked Questions

<Accordion title="How does zkPass work?">
  zkPass uses Three-Party TLS (3P-TLS) and Hybrid Zero-Knowledge Proofs for secure data handling.

  It begins with a three-party handshake between a data source (S), user (P), and zkPass node (V). Session keys are jointly derived such that the verifier cannot access private user data.

  zkPass combines interactive VOLE-based ZK proofs with non-interactive verification. Proof results are committed as zkSBTs for verifiable, privacy-preserving validation.

  [Technical Overview](https://medium.com/zkpass/a-technical-overview-of-zkpass-protocol-e28303e472e9)
</Accordion>

<Accordion title="How does zkPass prove the truth of HTTPS data?">
  zkPass relies on two guarantees:

  1. **Data integrity**\
     Any tampering breaks MAC verification in the three-party TLS session.
  2. **Statement correctness**\
     Proofs succeed only if the data satisfies verifier-defined conditions.
</Accordion>

<Accordion title="How is the credibility of data sources ensured?">
  During TLS handshake, zkPass independently verifies the server certificate and signature, ensuring the authenticity of the data source.
</Accordion>

<Accordion title="What are the key advantages of zkPass?">
  - Privacy-preserving (no raw data upload)
  - Verifiable via three-party TLS
  - Compatible with standard HTTPS websites
  - Anti-cheating protections
  - Fast local proof generation via VOLE-based ZK
</Accordion>

<Accordion title="Will zkPass TransGate handle or store private data?">
  No. TransGate runs entirely client-side.\
  Credentials are used only to retrieve data directly from the source, and proofs are generated locally.
</Accordion>

<Accordion title="How is local data tampering prevented?">
  zkPass verifies:

  - Correct encrypted requests
  - Valid request URLs
  - Authentic encrypted responses
  - Template-conforming attributes
  - Logical correctness of asserted values

  [Anti-cheating mechanisms](https://medium.com/zkpass/introducing-the-anti-cheating-mechanisms-on-zkpass-4427d24cbe7d)
</Accordion>

<Accordion title="How are duplicate accounts with different addresses handled?">
  A unique server-side identifier is hashed.\
  If a duplicate is detected, the newest attestation remains valid and older ones are invalidated.
</Accordion>

<Accordion title="How is the zkPass network distributed?">
  The network begins with zkPass-operated nodes and progressively decentralizes through economic incentives until equilibrium is reached.
</Accordion>

<Accordion title="Can I migrate attestations to a new wallet?">
  Minting the same schema from a new wallet invalidates the previous attestation.
</Accordion>
