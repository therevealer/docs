This document describes the zkPass oracle protocol from a technical perspective, detailing the cryptographic components and core building blocks used to construct the protocol.

The zkPass protocol is designed as a private data oracle, leveraging **Vector Oblivious Linear Evaluation (VOLE)** to construct efficient commitments, known as **VOLE-based Zero-Knowledge Proofs (VOLE-ZK)**. Beyond its use of SoftSpokenOT for Oblivious Transfer (OT), protocol security relies critically on the robustness of the **VOLE-in-the-Head (VOLEitH)** technique.

zkPass operates in **Hybrid Mode**, combining **Proxy Mode** and **MPC Mode**. This allows the protocol to adapt to different network conditions and server restrictions without compromising security or performance.

---

## Overview

While Transport Layer Security (TLS) provides secure access to private data, it does not allow users to prove to third parties that data genuinely originated from a specific website.

Existing approaches either introduce undesirable trust assumptions or require server-side modifications, effectively locking user data at its source.

The zkPass protocol addresses this limitation by extending TLS into **Three-Party TLS (3P-TLS)**, enabling secure, privacy-preserving data proofs without requiring cooperation or permission from the data holder.

The protocol involves three parties:

- **Prover (P)** – the data owner
- **Verifier (V)** – the proof consumer
- **Server (S)** – the TLS data source

The prover extracts data directly from the server using standard credentials, then produces a **Zero-Knowledge Proof (ZKP)** that the verifier can check without learning any private data.

To achieve this, zkPass integrates:

- Three-Party TLS (3P-TLS)
- Secure Multi-Party Computation (MPC)
- Non-Interactive Zero-Knowledge (NIZK)

## Operating Modes

zkPass supports two execution modes:

1. **Proxy Mode** – the verifier acts as a proxy
2. **MPC Mode** – prover and verifier jointly act as a client

These modes form **Hybrid Mode**, enabling seamless fallback when servers block requests from multiple IP addresses.

---

## Proxy Mode

In Proxy Mode, the verifier mediates communication between the prover and the server, recording TLS traffic for later verification.

### Protocol Flow

1. **Three-Party Handshake**  
   Prover, verifier, and server execute a 3P-TLS handshake.

2. **Prover Commitment**  
   The prover commits to their key share \( K_p \) before learning the full session key.

3. **Verifier Key Reveal**  
   The verifier reveals \( K_v \), allowing the prover to compute:
   \[
   K = K_p + K_v
   \]

4. **Session Execution**  
   The prover continues the TLS session while the verifier records traffic.

5. **Proof Generation**  
   The prover proves statements about the session transcript.

Server authenticity is verified via standard TLS certificate signatures. Privacy and integrity hold under standard TLS assumptions, with mild and practical network assumptions regarding routing attacks.

Proxy Mode achieves high performance by eliminating cryptographic overhead after the handshake.

## VOLEitH Protocol

VOLEitH is the most efficient client-side ZKP construction for TLS authorization.

The zkPass implementation combines:

- SoftSpokenOT (ROY22)
- VOLE-ZK (YSWW21)
- Fiat–Shamir Transform (FS87)

---

## VOLE Foundations

zkPass uses VOLE correlations over a finite field \( F_{2^k} \).

A VOLE correlation consists of:

- Global key \( \Delta \in F_{2^k} \)
- Values \( u_i, v_i \in F_{2^k} \)
- Tags \( q_i \in F_{2^k} \)

Defined as:
<!-- \[
q_i = u_i \cdot \Delta + v_i \quad \text{for } i = 0, \dots, \ell - 1
\]

The prover knows \( u_i, v_i \); the verifier knows \( q_i, \Delta \). -->

This acts as a linearly homomorphic commitment scheme that is:

- **Hiding** due to random masking
- **Binding** under the hardness of guessing \( \Delta \)

## VOLEitH Commitments

VOLEitH builds on **GGM-based vector commitments**, where the prover commits to \( N \) pseudo-random seeds using a single hash.

The prover can later open all but one seed with \( O(\log N) \) communication.

Seeds are expanded via a PRG into bit vectors \( r_i \), then aggregated:
\[
u = \sum r_i \quad,\quad v = \sum i \cdot r_i
\]

This construction yields VOLE correlations once the verifier learns all but one seed.

---

## VOLEitH Proof System

VOLE-ZK operates in the VOLE-hybrid model and follows the **Line–Point ZKP paradigm**.

zkPass uses the **QuickSilver** protocol to prove Boolean circuit satisfaction. Fiat–Shamir converts the interactive protocol into a NIZK.

Verification checks:
\[
\sum q_i \chi_i = \left(\sum u_i \chi_i\right) \cdot \Delta + \sum v_i \chi_i
\]

Correctness implies circuit satisfaction.

---

## Why VOLEitH

Traditional SNARKs are unsuitable for TLS authorization due to:

- Expensive cryptographic operations
- Large memory requirements
- Trusted setup assumptions

VOLE-ZK proofs are far more efficient but interactive.

VOLEitH resolves this by making VOLE-ZK **non-interactive**, enabling fast, browser-based proof generation without trusted setup.

## MPC Mode

MPC Mode jointly executes the TLS client using ECDH enhanced with MPC and OT.

### Three-Party Handshake

- Prover and verifier obtain additive shares of the pre-master key
- Server retains the full key
- Certificates and signatures ensure server authenticity

Session keys are derived via Garbled Circuits, ensuring:

- Verifier never sees encrypted data
- Tampering is detectable via MAC verification

Extensive optimizations reduce MPC overhead, including Silent OT, stacked garbled circuits, and optimized AES proofs.

---

## ZKP Phase

After TLS completion, the prover generates a VOLEitH-based NIZK proof for public verification.

This step mirrors Proxy Mode proof generation.

---

## Why MPC Is Secondary

MPC Mode incurs heavy overhead:

- Fixed 20MB session cost
- Large per-byte communication costs
- High computation requirements

It also introduces trust assumptions similar to proxy-based systems.

As a result, MPC Mode is reserved as a **fallback** for servers that block Proxy Mode.

---

## Summary

The zkPass protocol combines:

- Three-Party TLS
- VOLEitH-based NIZK
- Hybrid execution modes

This design enables secure, scalable, and privacy-preserving data verification without trusted servers or heavy cryptography.

Proxy Mode delivers high performance, while MPC Mode ensures robustness in restrictive environments. VOLEitH enables efficient non-interactive proofs, making zkPass practical for real-world deployment.
